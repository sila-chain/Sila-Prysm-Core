# Sila Proto Shadow Layer

Goal: introduce Sila-native protobuf identity while preserving Sila protobuf compatibility until all callers migrate safely.

## Current compatibility source

The current project still depends on:

- proto/sila/ext
- proto/sila/v1

These packages are used by:

- proto/silaengine/v1
- proto/sila/v1alpha1
- proto/migration
- beacon-chain packages
- rpc packages
- p2p packages

## Migration strategy

### Phase 1: Shadow ext package

Create Sila-native equivalent:

- proto/sila/ext

It must be functionally identical to proto/sila/ext at first.

Rules:

- no behavior change
- no generated API breakage
- no deletion of proto/sila/ext
- proto/sila/ext remains compatibility

### Phase 2: Shadow v1 package

Create Sila-native equivalent:

- proto/sila/v1

It must mirror proto/sila/v1 initially.

Rules:

- no message field changes
- no wire format changes
- no field number changes
- no JSON name changes

### Phase 3: Internal migration

Move internal imports gradually:

- proto/sila/v1 -> proto/sila/v1
- proto/sila/ext -> proto/sila/ext

Only after each package passes tests.

### Phase 4: Compatibility removal

Remove proto/sila only after:

- all internal imports are migrated
- generated files compile
- beacon-chain builds
- silactl builds
- validator builds
- RPC tests pass
- proto migration tests pass

## Hard rule

No broad replace across the repository. Every migration must be scoped, tested, committed, and pushed.

## Current blocker: protobuf generation pipeline

Sila protobuf shadow packages exist, but internal migration from proto/sila/v1 to proto/sila/v1 must not be done with manual protoc commands.

Reason:

- Sila protobuf generation is Bazel-based.
- proto/sila/v1/BUILD.bazel uses go_proto_library.
- It uses @com_github_sila_protoc_gen_go_cast//:go_cast_grpc.
- SSZ code is generated by ssz_gen_marshal.
- gateway.ssz.go depends on casted primitive types.
- Manual protoc --go_out breaks cast types.
- Manual protoc --go-cast_out with the locally installed plugin is not equivalent to the Bazel pipeline.
- The local Windows environment currently does not have bazel or bazelisk.

Next safe step:

Install or provide Bazel/Bazelisk, then build:

bazel build //proto/sila/v1:go_default_library

Only after the Bazel-generated Sila v1 output is verified should internal imports migrate from proto/sila/v1 to proto/sila/v1.
